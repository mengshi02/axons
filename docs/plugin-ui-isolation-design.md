# Axons Plugin UI iframe Isolation Design

> Version: v2.0 | Date: 2026-05-16 | Status: Implemented

## 1. Background and Motivation

### 1.1 Current Issues

Current plugin UI components are dynamically loaded into the main UI's React component tree via `import()`, sharing the same JS main thread with the main UI. This causes:

- **Plugin anomalies block the main UI**: Synchronous infinite loops, CPU-intensive computations, or infinite `setState` re-renders within plugin components can all block the entire JS event loop
- **ErrorBoundary cannot provide a safety net**: React's ErrorBoundary only catches throws during the render phase and cannot handle runtime blocking
- **DOM/CSS mutual pollution**: Plugin components share the DOM tree and CSS namespace with the main UI, making style conflicts hard to avoid
- **Plugin crashes affect globally**: A plugin's `pluginApi.fetch()` hanging without a timeout may leave the panel in an infinite loading state

### 1.2 Refactoring Window

There is currently only one plugin, making this the best time to refactor. Migration costs multiply as the number of plugins increases.

### 1.3 Solution Comparison

| Approach | Isolation Level | Refactoring Complexity | Blocking Resolution | Plugin Developer Impact |
|----------|----------------|----------------------|---------------------|------------------------|
| Short-term: Harden current architecture | No isolation | Low (2-3 days) | ~60% | None |
| **Mid-term: iframe isolation** | **Fault isolation (JS thread isolation)** | **Medium (6-7 days)** | **~95%** | **Low** |
| Long-term: Worker+iframe | Fault + Security dual isolation | High (3-4 weeks) | ~100% | Breaking Change |

Choosing the mid-term approach: iframe isolation achieves the best balance among isolation effectiveness, refactoring cost, and plugin developer impact.

> **⚠️ Important Distinction**: The iframe + `allow-same-origin` approach provides **fault isolation** (plugin JS anomalies will not block the main UI event loop), not **security isolation**. Since the iframe shares the same origin as the main UI and has `allow-same-origin` enabled, plugin JS can still access the main UI's DOM and global variables via `window.parent`. For security isolation (preventing malicious plugins), the `allow-same-origin` flag needs to be removed in the long-term solution, or a Worker + OOPIF approach should be adopted.

### 1.4 Implementation Principle: Thin Frontend, Thick Backend

This design follows the **thin frontend, thick backend** principle — the frontend only handles basic rendering and interaction; all logic involving performance and functionality is implemented in Go on the backend.

| Layer | Responsibility Boundary | What It Does NOT Do |
|-------|------------------------|---------------------|
| **Frontend (inside iframe)** | Render UI, make HTTP requests, subscribe to SSE, postMessage UI control signals | No event relay, no request proxying, no state management, no message deduplication |
| **Frontend (main UI)** | Manage iframe lifecycle, respond to UI control signals (close/ready) | No proxying API requests, no bridging EventBus, no runtime mode dispatching |
| **Backend (Go)** | EventBus broadcast center, API proxy, plugin state management, SSE push, CSP generation | — |

**Design Decision Mapping**:

| Decision Point | Thin Frontend / Thick Backend Choice | Rationale |
|---------------|-------------------------------------|-----------|
| PluginApi.fetch: direct or bridged? | **Desktop direct to plugin backend, Web via daemon proxy** | Consistent with pre-refactoring behavior, no degradation due to isolation |
| PluginApi.getState/setState: bridged? | **iframe fetches daemon API directly** | Same-origin HTTP request, frontend fetches directly, Go handles storage |
| EventBus event relay? | **daemon SSE broadcast** | Go serves as broadcast center, naturally loop-free; frontend only subscribes, no relay |
| Desktop/Web dispatching? | **Unified via daemon proxy** | Go proxy layer stream passthrough, performance identical to direct connection; eliminates frontend runtimeMode dispatch logic |
| CSP connect-src dynamic generation? | **Dynamically generated by runtime mode** | Desktop allows `http://127.0.0.1:*`, Web uses `'self'`; frontend does no dispatching, backend injects runtimeMode |
| theme.css dual-source? | **Single Source of Truth** | One theme.css, main UI references via @import, Go template references via link |
| postMessage responsibility? | **UI control signals only** | onClose / ready / error, not a data channel |

## 2. Four-Platform Compatibility Analysis

### 2.1 Prerequisite: Desktop Loading Mode

See the top comment in [`desktop/main.go`](../desktop/main.go) — Electron desktop webview directly loads the daemon's `http://127.0.0.1:<random-port>`, so **all three desktop platforms use same-origin HTTP**. There is no `file://` or custom protocol. This means the compatibility of iframe + sandbox + postMessage is equivalent to browser behavior under standard same-origin HTTP, and will not trigger historical compatibility issues of various WebViews under `file://`.

### 2.2 Platform WebView Compatibility Matrix

| Runtime Environment | Underlying Engine | iframe Rendering | sandbox Attribute | postMessage | Same-Origin fetch | CSP frame-ancestors | Risk Level |
|---------------------|-------------------|-----------------|-------------------|-------------|-------------------|---------------------|------------|
| macOS | WKWebView (Safari/WebKit) | ✅ | ✅ Full HTML5 standard set | ✅ | ✅ | ✅ | Very Low |
| Windows | WebView2 (Chromium Evergreen) | ✅ | ✅ Full HTML5 standard set | ✅ | ✅ | ✅ | Very Low |
| Linux | WebKitGTK 4.0/4.1 | ✅ | ✅ Full HTML5 standard set | ✅ | ✅ | ✅ | Low |
| Web Browser | Chromium/Firefox/Safari | ✅ | ✅ | ✅ | ✅ | ✅ | Very Low |

**Version Notes**:
- macOS: WKWebView follows the system Safari/WebKit. This project's minimum support is macOS 11+ (Safari 14+). HTML5 iframe sandbox has been fully supported since Safari 10+
- Windows: WebView2 uses Evergreen mode with automatic updates, keeping in sync with the latest Chromium
- Linux: WebKitGTK 4.0 (approaching EOL) and 4.1 both fully support the sandbox flags used in this design. The project's [`desktop/README.md`](../desktop/README.md) recommends installing 4.1 preferentially
- Web: Fully supported by modern browsers

### 2.3 Key Risks and Mitigations

#### Risk 1: `window.close()` may be blocked in WKWebView

WKWebView, for security reasons, does not respond to `window.close()` calls on windows not opened by script by default.

**Mitigation**: The `onClose` callback does **not** use `window.close()`. Instead, it sends a close request to the main UI via `postMessage('plugin:close')`, and the main UI removes the iframe DOM upon receipt. This approach works universally across all four platforms and does not depend on any sandbox flag.

#### Risk 2: iframe cannot inherit main UI CSS variables

An iframe is an independent browsing context and cannot inherit CSS custom properties from the parent page.

**Mitigation**:
1. Inside the iframe, load theme CSS via `<link href="/plugin-sdk/theme.css">` (same-origin loading with the main UI, zero network overhead)
2. In [`ui/src/plugin-sdk/theme.css`](../ui/src/plugin-sdk/theme.css), explicitly declare CSS variables for `:root.moon-theme` and `:root.sun-theme` (not relying on parent-level inheritance)
3. When the main UI theme changes, notify the iframe via `postMessage('plugin:theme', { theme })`
4. Inside the iframe, JS toggles the `:root` class (moon-theme/sun-theme), and CSS variables automatically follow

#### Risk 3: postMessage communication latency

Each cross-iframe postMessage has approximately 1-5ms overhead, which may impact performance in high-frequency scenarios.

**Mitigation**: The PluginApi implementation layer dispatches by operation frequency:
- **High-frequency operations** (`fetch`/`createEventSource`) → iframe directly connects to plugin backend, no postMessage
- **Low-frequency operations** (`onEvent`/`emitEvent`/`getState`/`setState`) → postMessage bridging

#### Risk 4: WebKitGTK differences across Linux distributions

Some older Linux distributions may still ship with WebKitGTK 4.0 (EOL date 2025). There is no functional difference from 4.1 for the features used in this design, but it is recommended to run a regression on both 4.0 and 4.1 environments during the PoC phase.

### 2.4 Verification Strategy

To eliminate all uncertainties, before implementation, run a minimal PoC on macOS / Windows / Linux (see Section 8 Implementation Plan) to verify four items: iframe rendering, sandbox enforcement, bidirectional postMessage communication, and same-origin fetch passthrough.

### 3.2 PluginApi Communication Architecture (Thin Frontend, Thick Backend)

**Core Principle**: The iframe is an independent web application that communicates with the daemon via standard HTTP/SSE. postMessage is only used for UI coordination between parent and child windows.

```
PluginApi Interface (signatures unchanged)
├── fetch(path, opts)         → Desktop: direct to endpoint+path; Web: via /v1/plugins/:id/proxy (consistent with pre-refactoring behavior)
├── createEventSource(path)   → Desktop: direct to endpoint+path; Web: via /v1/plugins/:id/proxy (consistent with pre-refactoring behavior)
├── getState(key)             → iframe fetch → /v1/plugins/state/:id:key → daemon state storage
├── setState(key, value)      → iframe fetch PUT → /v1/plugins/state/:id:key → daemon state storage
├── emitEvent(type, payload)  → iframe fetch POST → /v1/plugins/event → daemon EventBus
├── onEvent(type, handler)    → iframe EventSource → /v1/plugins/events/stream → daemon SSE broadcast
└── onClose()                 → postMessage('plugin:close') → main UI removes iframe (sole postMessage usage)
```

**Why all via iframe HTTP direct connection?**

1. **getState/setState are already HTTP APIs** — iframe same-origin can directly fetch, no need for postMessage detour through the main UI
2. **daemon is the natural EventBus broadcast center** — iframe subscribes via SSE, naturally loop-free (daemon is the single broadcast source)
3. **Thin frontend layer = simple and reliable** — no Bridge.handleApiRequest, no pendingRequests, no AbortController, no messageId deduplication
4. **Thick Go backend = high performance** — daemon uses in-memory EventBus + SSE push, performance is tens of times higher than frontend JS relay

> **Desktop/Web fetch dispatching rule (unchanged)**: `fetch` / `createEventSource` directly connect to the plugin backend on desktop (zero proxy overhead), and go through the daemon proxy on Web. The iframe-adapter determines the runtime mode via `window.__AXONS_PLUGIN__.runtimeMode` for dispatching, which is fully consistent with the pre-refactoring [`pluginApi.ts`](../ui/src/lib/pluginApi.ts) behavior. The other methods (getState/setState/emitEvent/onEvent) are daemon's own APIs with no dispatching needed; they uniformly use same-origin HTTP/SSE.
>
> **CSP Note**: When directly connecting to the plugin backend port on desktop, CSP `connect-src` needs to allow `http://127.0.0.1:*` (or a specific port). Therefore, CSP must be dynamically generated based on the runtime environment — desktop: `connect-src 'self http://127.0.0.1:*'`, Web: `connect-src 'self'`. The daemon template injects `runtimeMode` for iframe-adapter dispatching.

## 4. postMessage Communication Protocol

> **Thin Frontend, Thick Backend Design**: postMessage is only used for UI control signals (close/ready/theme); data channels all go through HTTP/SSE direct to daemon. Message types are simplified from the original 10 to 4.

### 4.1 Message Format

All postMessage messages use a unified envelope format:

```typescript
interface PluginMessage {
  /** Message protocol identifier */
  protocol: 'axons-plugin-iframe';
  /** Protocol version number — for future dual-track upgrades */
  version: 1;
  /** Message source: 'host' | 'plugin' */
  source: 'host' | 'plugin';
  /** Plugin ID */
  pluginId: string;
  /** Message type */
  type: MessageType;
  /** Message payload */
  payload?: any;
}
```

**Version Negotiation Rules**: Currently `version: 1`. For future protocol upgrades, both host and plugin should silently ignore unknown versions (no errors thrown) and maintain forward compatibility.

### 4.2 Origin Verification

Since the iframe and main UI are loaded from the same origin (see Section 2.1), bidirectional postMessage must perform origin verification:

- **Sender**: `postMessage(msg, window.location.origin)` — `targetOrigin` is fixed to the current origin, `'*'` is prohibited
- **Receiver**: `if (event.origin !== window.location.origin) return` — immediately discard if origin does not match

### 4.3 Message Types (4 types)

```typescript
type MessageType =
  // ---- Main UI → iframe ----
  | 'plugin:init'           // Initialization: inject pluginId (needed for HTTP requests inside iframe)
  | 'plugin:theme'          // Theme change: { theme: 'moon' | 'sun' }

  // ---- iframe → Main UI ----
  | 'plugin:ready'          // iframe loaded
  | 'plugin:close'          // Request to close panel
```

**Differences from v1 design**:

| Removed Message Type | Original Purpose | Current Replacement |
|---------------------|------------------|---------------------|
| `plugin:api-request` | getState/setState via postMessage | iframe directly fetches daemon API |
| `plugin:api-response` | Return API results | Not needed, iframe directly handles HTTP response |
| `plugin:event` (iframe→host) | emitEvent via postMessage | iframe POSTs `/v1/plugins/event` directly |
| `plugin:event` (host→iframe) | onEvent via postMessage bridging | iframe subscribes to daemon SSE via EventSource |
| `plugin:resize` | Notify panel size change | iframe monitors `window.resize` itself |
| `plugin:error` | Plugin error reporting | iframe handles `window.onerror` itself |

## 6. Detailed Implementation

### 6.1 IframePluginPanel Component

```tsx
// ui/src/components/IframePluginPanel.tsx
// Thin frontend, thick backend: main UI side only manages iframe lifecycle + theme notification, no data bridging

interface IframePluginPanelProps {
  def: PanelDef;
  onClose: () => void;
}

function IframePluginPanel({ def, onClose }: IframePluginPanelProps) {
  const iframeRef = useRef<HTMLIFrameElement>(null);
  const [panelWidth, setPanelWidth] = useState(DEFAULT_WIDTH);
  const [iframeReady, setIframeReady] = useState(false);
  const [iframeError, setIframeError] = useState<string | null>(null);

  // Listen for iframe postMessage (UI control signals only: ready/close)
  useEffect(() => {
    const handleMessage = (event: MessageEvent) => {
      if (event.origin !== window.location.origin) return;
      const msg = event.data;
      if (msg?.protocol !== 'axons-plugin-iframe' || msg.version !== 1) return;
      if (msg.pluginId !== (def.pluginId || '')) return;

      switch (msg.type) {
        case 'plugin:ready':
          setIframeReady(true);
          break;
        case 'plugin:close':
          onClose();
          break;
      }
    };
    window.addEventListener('message', handleMessage);
    return () => window.removeEventListener('message', handleMessage);
  }, [def.pluginId, onClose]);

  // Theme change notification (only proactively sent postMessage)
  const { theme } = useTheme();
  useEffect(() => {
    if (!iframeRef.current?.contentWindow || !iframeReady) return;
    iframeRef.current.contentWindow.postMessage({
      protocol: 'axons-plugin-iframe',
      version: 1,
      source: 'host',
      pluginId: def.pluginId || '',
      type: 'plugin:theme',
      payload: { theme },
    }, window.location.origin);
  }, [theme, iframeReady]);

  return (
    <div className={containerClass} style={{ width: panelWidth }}>
      <div className={resizeHandleClass} onMouseDown={handleResizeMouseDown} />
      {!iframeReady && !iframeError && <Spinner />}
      {iframeError && <ErrorDisplay error={iframeError} onRetry={...} />}
      <iframe
        ref={iframeRef}
        src={`/v1/plugins/${def.pluginId}/iframe-host`}
        sandbox="allow-same-origin allow-scripts allow-forms allow-modals"
        className={iframeReady ? 'w-full h-full border-0' : 'hidden'}
        onError={() => setIframeError('Plugin failed to load')}
      />
    </div>
  );
}
```

### 6.2 PluginIframeBridge No Longer Needed

In the v1 design, `PluginIframeBridge` handled API request proxying, EventBus event bridging, and request-response matching (~120 lines). In v2, these data channels are all moved to the backend. The main UI only needs to listen for `message` events within `IframePluginPanel` (~20 lines), without an independent Bridge class.

**Removed Complexity**:
- `pendingRequests` Map + requestId matching → Not needed
- `AbortController` + interrupt fetch on destroy → Not needed
- `messageId` deduplication LRU cache → Not needed
- `handleApiRequest` (getState/setState proxy) → Not needed
- `forwardEvent` (EventBus bridging) → Not needed

### 6.3 iframe-adapter (Plugin SDK Side, Runs Inside iframe)

> **Core Manifestation of Thin Frontend, Thick Backend**: All PluginApi methods go through HTTP/SSE direct to daemon; postMessage is only used for UI control.

```typescript
// ui/src/plugin-sdk/iframe-adapter.ts

/**
 * iframe-adapter — Runs inside the plugin iframe, wrapping HTTP/SSE as PluginApi interface.
 * Thin frontend, thick backend: data channels all directly connect to daemon, UI control uses postMessage.
 * Plugin developers do not need to use this module directly; it is automatically loaded by the daemon iframe-host template.
 */

export class IframePluginApiAdapter implements PluginApi {
  private static readonly PROTOCOL_VERSION = 1;
  private readonly expectedOrigin = window.location.origin;
  private _pluginId = '';
  private eventSource: EventSource | null = null;
  private eventHandlers = new Map<string, Set<(payload: any) => void>>();

  constructor() {
    window.addEventListener('message', this.handleMessage);
  }

  /** Wait for the main UI to send init message, initialize pluginId */
  init(): Promise<{ pluginId: string }> {
    return new Promise((resolve) => {
      const handler = (event: MessageEvent) => {
        if (event.origin !== this.expectedOrigin) return;
        const msg = event.data;
        if (msg?.protocol !== 'axons-plugin-iframe' || msg.version !== IframePluginApiAdapter.PROTOCOL_VERSION) return;
        if (msg.type !== 'plugin:init') return;
        this._pluginId = msg.payload.pluginId;
        window.removeEventListener('message', handler);
        this.send({ type: 'plugin:ready' });
        resolve({ pluginId: this._pluginId });
      };
      window.addEventListener('message', handler);
    });
  }

  // ---- PluginApi interface implementation (all via HTTP/SSE direct to daemon) ----

  get pluginId(): string { return this._pluginId; }

  /** fetch → Desktop: direct to plugin backend; Web: via daemon proxy (consistent with pre-refactoring behavior) */
  async fetch(path: string, opts?: RequestInit): Promise<Response> {
    return globalThis.fetch(this.resolveUrl(path), opts);
  }

  /** createEventSource → Desktop: direct to plugin backend; Web: via daemon proxy */
  createEventSource(path: string): EventSource {
    return new EventSource(this.resolveUrl(path));
  }

  /** getState → iframe directly connects to daemon state API */
  async getState(key: string): Promise<any> {
    const resp = await globalThis.fetch(`/v1/plugins/state/${this.pluginId}:${key}`);
    if (!resp.ok) return null;
    const data = await resp.json();
    return data.value;
  }

  /** setState → iframe directly connects to daemon state API */
  async setState(key: string, value: any): Promise<void> {
    await globalThis.fetch(`/v1/plugins/state/${this.pluginId}:${key}`, {
      method: 'PUT',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify(value),
    });
  }

  /** onEvent → iframe directly connects to daemon SSE broadcast */
  onEvent(type: string, handler: (payload: any) => void): () => void {
    if (!this.eventHandlers.has(type)) {
      this.eventHandlers.set(type, new Set());
    }
    this.eventHandlers.get(type)!.add(handler);

    // Create SSE connection on first subscription
    if (!this.eventSource) {
      this.connectSSE();
    }

    return () => {
      this.eventHandlers.get(type)?.delete(handler);
    };
  }

  /** emitEvent → iframe directly POSTs to daemon EventBus */
  async emitEvent(type: string, payload: any): Promise<void> {
    await globalThis.fetch('/v1/plugins/event', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ pluginId: this.pluginId, type, payload }),
    });
  }

  /** onClose → postMessage to main UI (sole postMessage usage) */
  onClose(): void {
    this.send({ type: 'plugin:close' });
  }

  // ---- SSE Connection Management ----

  /** resolveUrl — Desktop: direct to plugin backend; Web: via daemon proxy (consistent with pre-refactoring pluginApi.ts behavior) */
  private resolveUrl(path: string): string {
    if (window.__AXONS_PLUGIN__?.runtimeMode === 'desktop' && window.__AXONS_PLUGIN__?.endpoint) {
      return window.__AXONS_PLUGIN__.endpoint + path;
    }
    return `/v1/plugins/${this.pluginId}/proxy${path}`;
  }

  private connectSSE() {
    this.eventSource = new EventSource('/v1/plugins/events/stream');
    this.eventSource.onmessage = (event) => {
      try {
        const data = JSON.parse(event.data);
        // Broadcast all event types to local handlers
        const handlers = this.eventHandlers.get(data.type);
        if (handlers) {
          handlers.forEach(h => {
            try { h(data.payload); } catch (e) { console.error('[Plugin] Event handler error:', e); }
          });
        }
      } catch (e) { console.error('[Plugin] SSE parse error:', e); }
    };
  }

  // ---- postMessage UI Control ----

  private handleMessage = (event: MessageEvent) => {
    if (event.origin !== this.expectedOrigin) return;
    const msg = event.data;
    if (msg?.protocol !== 'axons-plugin-iframe' || msg.source !== 'host') return;
    if (msg.version !== IframePluginApiAdapter.PROTOCOL_VERSION) return;

    switch (msg.type) {
      case 'plugin:theme': {
        const root = document.documentElement;
        root.classList.remove('moon-theme', 'sun-theme');
        root.classList.add(msg.payload.theme === 'moon' ? 'moon-theme' : 'sun-theme');
        break;
      }
    }
  };

  private send(msg: Omit<PluginMessage, 'protocol' | 'version' | 'source' | 'pluginId'>) {
    window.parent.postMessage({
      protocol: 'axons-plugin-iframe',
      version: IframePluginApiAdapter.PROTOCOL_VERSION,
      source: 'plugin',
      pluginId: this.pluginId,
      ...msg,
    }, this.expectedOrigin);
  }
}
```

### 6.4 iframe Container HTML

In Phase 1, all plugin iframe containers are uniformly dynamically generated by the daemon via `/v1/plugins/:id/iframe-host` (see 6.7.2). Plugin developers **do not** need to write `index.html`. The daemon template automatically injects: theme.css, components.css, iframe-adapter.umd.js, axons-plugin-ui.umd.js, plugin `ui/index.js`, and `window.__AXONS_PLUGIN__ = { pluginId, protocolVersion }`.

However, a secure nonce injection mechanism must be designed (daemon performs secondary template processing on static HTML) to ensure the CSP security model is not bypassed. This capability is not enabled in Phase 1.

### 6.6 EventBus — daemon SSE Broadcast (Backend Thick Layer)

**Thin Frontend, Thick Backend Design**: The daemon serves as the EventBus broadcast center. Both iframes and built-in panels communicate with the daemon via HTTP/SSE, eliminating the need for main UI frontend JS bridge relay.

```
Built-in panel emit ──→ POST /v1/plugins/event ──→ daemon EventBus ──→ SSE broadcast
iframe emit ─────────→ POST /v1/plugins/event ──→ daemon EventBus ──→ SSE broadcast
                                                              ↕
                                              EventSource /v1/plugins/events/stream
                                                        ↕        ↕
                                                  Built-in panel   iframe
```

**Go Implementation**:

```go
// internal/plugin/eventbus.go

// PluginEventBus — Go in-memory EventBus, high-performance broadcast center
type PluginEventBus struct {
    mu      sync.RWMutex
    sinks   map[chan Event]struct{}  // SSE subscribers
}

type Event struct {
    PluginID string `json:"pluginId"`
    Type     string `json:"type"`
    Payload  any    `json:"payload"`
}

var globalBus = &PluginEventBus{
    sinks: make(map[chan Event]struct{}),
}

// HandlePostEvent — POST /v1/plugins/event
// Receives events from built-in panels or iframes, broadcasts to all SSE subscribers
func (b *PluginEventBus) HandlePostEvent(w http.ResponseWriter, r *http.Request) {
    var event Event
    if err := json.NewDecoder(r.Body).Decode(&event); err != nil {
        http.Error(w, "invalid event", http.StatusBadRequest)
        return
    }
    b.broadcast(event)
    w.WriteHeader(http.StatusNoContent)
}

// HandleEventStream — SSE /v1/plugins/events/stream
// Both built-in panels and iframes subscribe to events via this endpoint
func (b *PluginEventBus) HandleEventStream(w http.ResponseWriter, r *http.Request) {
    flusher, ok := w.(http.Flusher)
    if !ok {
        http.Error(w, "streaming not supported", http.StatusInternalServerError)
        return
    }

    ch := make(chan Event, 64)
    b.register(ch)
    defer b.unregister(ch)

    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache")
    w.Header().Set("Connection", "keep-alive")

    for {
        select {
        case event := <-ch:
            data, _ := json.Marshal(event)
            fmt.Fprintf(w, "data: %s\n\n", data)
            flusher.Flush()
        case <-r.Context().Done():
            return
        }
    }
}

func (b *PluginEventBus) broadcast(event Event) {
    b.mu.RLock()
    defer b.mu.RUnlock()
    for ch := range b.sinks {
        select {
        case ch <- event:
        default:
            // Skip slow consumers to avoid blocking broadcast
        }
    }
}

func (b *PluginEventBus) register(ch chan Event) {
    b.mu.Lock()
    defer b.mu.Unlock()
    b.sinks[ch] = struct{}{}
}

func (b *PluginEventBus) unregister(ch chan Event) {
    b.mu.Lock()
    defer b.mu.Unlock()
    delete(b.sinks, ch)
    close(ch)
}
```

**Built-in Panel Adaptation**: The main UI's `pluginEventBus.ts` is changed to send events via `POST /v1/plugins/event` and subscribe to events via `EventSource('/v1/plugins/events/stream')`, using the same channel as iframes. Bridge methods like `registerBridge`/`forwardToIframe` are all removed.

### 6.7 Backend Changes

The backend involves three changes: ① Fix the existing static file path traversal vulnerability; ② Add `HandlePluginIframeHost` route for dynamically generating iframe container HTML; ③ Add EventBus SSE broadcast endpoints.

#### 6.7.1 Fix HandlePluginStaticFiles Path Traversal Vulnerability

The current [`internal/plugin/proxy.go`](../internal/plugin/proxy.go:108) directly concatenates `pluginDir + filePath`, allowing `../../../etc/passwd` to potentially escape the plugin directory. This should be fixed during the refactoring:

```go
// internal/plugin/proxy.go — HandlePluginStaticFiles refactoring

func (m *Manager) HandlePluginStaticFiles(w http.ResponseWriter, r *http.Request, pluginID string, filePath string) {
    // ... existing plugin lookup logic ...

    // Security: clean path and verify it does not escape the plugin directory
    cleanPath := filepath.Clean("/" + filePath)
    fullPath := filepath.Join(pluginDir, cleanPath)
    absPluginDir, err := filepath.Abs(pluginDir)
    if err != nil {
        http.Error(w, "internal error", http.StatusInternalServerError)
        return
    }
    absFullPath, err := filepath.Abs(fullPath)
    if err != nil || !strings.HasPrefix(absFullPath, absPluginDir+string(os.PathSeparator)) {
        http.Error(w, "forbidden", http.StatusForbidden)
        return
    }

    // Allow same-origin iframe to embed plugin static resources
    w.Header().Set("Content-Security-Policy", "frame-ancestors 'self'")

    http.ServeFile(w, r, absFullPath)
}
```

#### 6.7.2 Add HandlePluginIframeHost — Dynamically Generate iframe Container

The daemon provides a unified iframe container route `/v1/plugins/:id/iframe-host`, dynamically injecting `pluginId` and CSP nonce:

```go
// internal/plugin/proxy.go — new addition

const iframeHostTemplate = `<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>{{.PluginID}}</title>
  <link rel="stylesheet" href="/plugin-sdk/theme.css" />
  <link rel="stylesheet" href="/plugin-sdk/components.css" />
  <style>
    body { margin: 0; padding: 0; overflow: hidden; background: var(--axons-color-surface, #101018); }
  </style>
</head>
<body>
  <div id="root"></div>
  <script nonce="{{.Nonce}}">
    window.__AXONS_PLUGIN__ = {
      pluginId: {{.PluginIDJSON}},
      endpoint: {{.EndpointJSON}},
      runtimeMode: {{.RuntimeModeJSON}},
      protocolVersion: 1
    };
  </script>
  <script nonce="{{.Nonce}}" src="/plugin-sdk/iframe-adapter.umd.js"></script>
  <script nonce="{{.Nonce}}" src="/plugin-sdk/axons-plugin-ui.umd.js"></script>
  <script nonce="{{.Nonce}}" src="/plugins/{{.PluginID}}/ui/index.js"></script>
</body>
</html>`

// HandlePluginIframeHost renders the iframe container HTML and sets CSP.
// Route: GET /v1/plugins/:id/iframe-host
func (m *Manager) HandlePluginIframeHost(w http.ResponseWriter, r *http.Request, pluginID string) {
    inst, ok := m.GetInstance(pluginID)
    var endpoint string
    if ok && inst.Backend != nil {
        endpoint = inst.Backend.URL()
    }

    // Validate pluginID is legal (prevent template injection)
    if !validPluginID.MatchString(pluginID) {
        http.Error(w, "invalid plugin id", http.StatusBadRequest)
        return
    }

    // Determine runtime mode: desktop vs Web
    runtimeMode := "web"
    if getRuntimeMode() == "desktop" {
        runtimeMode = "desktop"
    }

    // Generate one-time nonce
    nonceBytes := make([]byte, 16)
    if _, err := rand.Read(nonceBytes); err != nil {
        http.Error(w, "internal error", http.StatusInternalServerError)
        return
    }
    nonce := base64.StdEncoding.EncodeToString(nonceBytes)

    // CSP — Desktop fetch direct to plugin backend needs to allow local ports
    // style-src includes 'unsafe-inline' as Phase 1 compromise (CSS-in-JS compatibility), Phase 2 will use nonce
    connectSrc := "'self'"
    if runtimeMode == "desktop" {
        connectSrc = "'self http://127.0.0.1:*'"
    }
    csp := fmt.Sprintf(
        "default-src 'self'; "+
            "script-src 'self' 'nonce-%s'; "+
            "style-src 'self' 'unsafe-inline'; "+
            "img-src 'self' data:; "+
            "connect-src %s; "+
            "frame-ancestors 'self'",
        nonce, connectSrc,
    )
    w.Header().Set("Content-Security-Policy", csp)
    w.Header().Set("Content-Type", "text/html; charset=utf-8")
    w.Header().Set("X-Content-Type-Options", "nosniff")

    pluginIDJSON, _ := json.Marshal(pluginID)
    endpointJSON, _ := json.Marshal(endpoint)
    runtimeModeJSON, _ := json.Marshal(runtimeMode)

    tmpl := template.Must(template.New("host").Parse(iframeHostTemplate))
    _ = tmpl.Execute(w, map[string]any{
        "PluginID":        pluginID,
        "Nonce":           nonce,
        "PluginIDJSON":    template.JS(pluginIDJSON),
        "EndpointJSON":    template.JS(endpointJSON),
        "RuntimeModeJSON": template.JS(runtimeModeJSON),
    })
}
```

> **Difference from v1**: Removed `endpoint` / `runtimeMode` injection (no longer needed for desktop dispatching) and `connect-src` dynamic generation (unified via proxy, static `'self'` suffices). The Go backend is not aware of the frontend runtime mode.

#### 6.7.4 Desktop Notes

See the architecture comment in [`desktop/main.go`](../desktop/main.go). Electron desktop webview loads the daemon's `http://127.0.0.1:<random-port>`. The main UI and iframe-host are served by the same daemon HTTP server:

- `frame-ancestors 'self'` works on all three desktop platforms
- Desktop `fetch`/`EventSource` directly connects to the plugin backend port; CSP `connect-src` dynamically allows `http://127.0.0.1:*`; Web uniformly uses proxy, `connect-src 'self'` suffices
- The daemon proxy layer uses `httputil.ReverseProxy` for stream passthrough; SSE latency is identical to direct connection

### 6.8 theme.css — Single Source of Truth

An iframe is an independent browsing context and cannot inherit parent page CSS variables. Theme CSS must be reloaded inside the iframe. However, "reloading" does not mean "re-declaring" — `theme.css` is the single variable source; both the main UI and iframe are its consumers.

**Architecture**:

```
plugin-sdk/theme.css  ←  Single Source of Truth (sole variable declaration location)
    ↑ @import               ↑ <link>
Main UI index.css          daemon iframe-host template
```

**Main UI Adaptation** (at the top of `ui/src/index.css`):

```css
/* Main UI references the same theme.css, no longer re-declares color variables */
@import './plugin-sdk/theme.css';
```

### 6.9 Long-Term Evolution

This design focuses on iframe isolation + thin frontend, thick backend architecture. Long-term evolution directions:

1. **Worker + iframe dual isolation**: Remove `allow-same-origin`, use Worker for JS computation isolation + iframe for DOM isolation, achieving fault + security dual isolation
2. **Binary stream support**: Upgrade daemon SSE to WebSocket, supporting bidirectional streaming communication
3. **htmlPath custom entry**: Enable in Phase 2 on demand, daemon performs secondary template processing on static HTML to inject CSP nonce

> The above are directional notes only, not implemented in this phase. Protocol version number `version: 1` is in place for future dual-track transition during upgrades.

## 7. Data Flows

### 7.1 Plugin Panel Open Flow

```
1. User clicks plugin panel button
2. App.tsx renders <IframePluginPanel def={panelDef}>
3. IframePluginPanel creates <iframe src="/v1/plugins/:id/iframe-host">
4. daemon returns iframe-host HTML, containing CSP nonce + injected window.__AXONS_PLUGIN__
5. iframe loads iframe-adapter.umd.js + axons-plugin-ui.umd.js + plugin index.js
6. iframe-adapter calls adapter.init() waiting for main UI's plugin:init message
7. IframePluginPanel sends plugin:init { pluginId } after listening for iframe load event
8. iframe-adapter initialization complete → sends plugin:ready
9. Main UI receives ready → iframeReady = true
10. iframe React component rendering complete
```

### 7.2 Plugin API Call Flows

```
fetch('/api/models'):
  Desktop: iframe → resolveUrl → endpoint + '/api/models'
                    → globalThis.fetch (direct to plugin backend, zero proxy overhead)
  Web:     iframe → resolveUrl → /v1/plugins/:id/proxy/api/models
                    → globalThis.fetch (same-origin, no CORS)
                    → axons daemon proxy → plugin backend

getState('config'):
  iframe → globalThis.fetch /v1/plugins/state/:id:config
         → daemon state storage → returns JSON

emitEvent('node:selected', { nodeId }):
  iframe → globalThis.fetch POST /v1/plugins/event
         → daemon EventBus → SSE broadcast to all subscribers

onEvent('node:selected'):
  iframe → new EventSource('/v1/plugins/events/stream')
         → daemon SSE push → iframe handler invoked
```

### 7.3 Theme Switch Flow

```
1. User switches theme (moon → sun)
2. useTheme updates → IframePluginPanel detects theme change
3. IframePluginPanel sends postMessage 'plugin:theme' to iframe
4. iframe-adapter receives → toggles :root class
5. CSS variables automatically update → plugin UI styles follow
```

### D. sandbox Attribute Explanation

```html
<iframe sandbox="allow-same-origin allow-scripts allow-forms allow-modals">
```

| Flag | Purpose | Required |
|------|---------|----------|
| `allow-same-origin` | Allows iframe to maintain same origin, can access same-origin APIs | ✅ Required — otherwise fetch/postMessage won't work |
| `allow-scripts` | Allows JavaScript execution | ✅ Required — plugin UI needs JS |
| `allow-forms` | Allows form submission | ✅ Recommended — plugins may have search forms |
| `allow-modals` | Allows alert/confirm/prompt | Optional — for debugging |
| `allow-popups` | Allows window.open | ❌ Not added — plugins should not open new windows |
| `allow-popups-to-escape-sandbox` | Allows popups to escape sandbox | ❌ Not added — does not depend on this flag |
